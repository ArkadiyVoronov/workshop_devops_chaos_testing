### **Часть 1: Теория Chaos Engineering (5 минут)**

#### Слайд 3: Что такое Chaos Engineering?
```
🌀 Chaos Engineering = "Умышленное введение сбоев"

Принципы Netflix (2011):
1. **Гипотеза:** Система устойчива к сбоям
2. **Эксперимент:** Вводим контролируемый сбой
3. **Наблюдение:** Измеряем влияние на систему
4. **Улучшение:** Устраняем уязвимости

🎯 Цель: Не "если" сломается, а "что делать когда сломается"
```

#### Слайд 4: Почему это важно для Финтех?
```
💳 Black Friday 2025 - 500M+ транзакций
⏱️ 1 минута простоя = $1M+ потерь
👥 1% недовольных клиентов = репутационные потери

Реальные примеры:
- PayPal 2010: $10M/час за downtime
- Robinhood 2020: $1B+ убытков от сбоев
- Наш случай: 2 часа простоя = потеря 5% клиентской базы
```

**Что говорить:**
"Chaos Engineering - это не про то, чтобы сломать систему, а про то, чтобы понять, как она поведет себя в реальных условиях. В финтехе 1 минута простоя может стоить миллионов. Сегодня мы покажем, как мы готовимся к Black Friday через контролируемые сбои."

### **Часть 3: Практическая демонстрация (30-40 минут)**

### **Кейс 1: Сбой базы данных (5-7 минут)**

#### Слайд 7: Кейс 1 - Database Failure
```
💥 Сценарий: Payment DB недоступна
📊 Ожидаемое воздействие:
- Невозможность обработки платежей
- Очереди транзакций
- Cascade failure на API

🎯 Что тестируем:
1. Graceful degradation
2. Retry mechanisms
3. Fallback strategies
4. Monitoring alerts
```

**Демонстрация (live):**
1. **Показать текущее состояние** в Grafana
2. **Запустить сбой:** `docker stop db` или `kill -9 <db_pid>`
3. **Наблюдать:**
   - Payment API возвращает 503
   - Очереди транзакций растут
   - Alerts в Grafana
4. **Обсудить:** "Что происходит? Как пользователи это видят?"

#### Слайд 8: Тушение пожара (Database Failure)
```
🔥 Что сломалось:
- Payment API не может записывать транзакции
- Frontend показывает ошибки
- Очереди растут

🚒 Как тушим:
1. **Immediate:** Read-only mode для API
2. **Short-term:** Переключение на replica DB
3. **Long-term:** Auto-failover в Kubernetes

📝 Lessons learned:
- Нужно улучшить connection pooling
- Добавить circuit breaker pattern
- Увеличить количество replicas
```

**Что говорить:**
"Видите, payment API падает, очереди растут. Пользователи видят ошибки. Теперь тушим пожар: включаем read-only режим, переключаемся на replica, активируем failover. Но это реактивно. Лучше предотвратить."

### **Кейс 2: Нагрузка на API Gateway (5-7 минут)**

#### Слайд 9: Кейс 2 - API Gateway Overload
```
💥 Сценарий: 10x рост трафика на Gateway
📊 Ожидаемое воздействие:
- Rate limiting срабатывает
- 429 ошибки для пользователей
- Cascade throttling на backend

🎯 Что тестируем:
1. Rate limiting effectiveness
2. Backpressure handling
3. Load balancing
4. Monitoring thresholds
```

**Демонстрация:**
1. **Запустить нагрузку:** `wrk -t12 -c400 -d30s http://localhost:5000`
2. **Показать в Grafana:** рост запросов, ошибки 429
3. **Наблюдать:** как система реагирует

#### Слайд 10: Тушение пожара (API Gateway Overload)
```
🔥 Что сломалось:
- 80% запросов получают 429
- Backend получает меньше нагрузки
- Пользователи видят "Сервис занят"

🚒 Как тушим:
1. **Immediate:** Увеличить rate limits
2. **Short-term:** Horizontal scaling Gateway
3. **Long-term:** Implement adaptive throttling

📝 Lessons learned:
- Rate limits слишком агрессивные
- Нужно dynamic scaling
- Добавить queue management
```

### **Кейс 3: Сбой аналитики (5-7 минут)**

#### Слайд 11: Кейс 3 - Analytics Service Failure
```
💥 Сценарий: ClickHouse кластер недоступен
📊 Ожидаемое воздействие:
- Аналитика не работает
- Dashboard пустые
- Business intelligence задержки

🎯 Что тестируем:
1. Graceful degradation аналитики
2. Caching strategies
3. Data consistency
4. Recovery time objective
```

**Демонстрация:**
1. **Остановить аналитику:** `docker stop analytics`
2. **Показать:** как система продолжает работать без аналитики
3. **Восстановить** и посмотреть recovery time

#### Слайд 12: Тушение пожара (Analytics Failure)
```
🔥 Что сломалось:
- Dashboard показывает "No data"
- Business users нервничают
- Reports не генерируются

🚒 Как тушим:
1. **Immediate:** Switch to cached data
2. **Short-term:** Manual failover to backup cluster
3. **Long-term:** Implement data replication

📝 Lessons learned:
- Нужен caching layer
- Replication lag monitoring
- Backup recovery testing
```

### **Кейс 4: Сетевой сбой (5-7 минут)**

#### Слайд 13: Кейс 4 - Network Partition
```
💥 Сценарий: Сбой между frontend и backend
📊 Ожидаемое воздействие:
- Frontend не может достучаться до API
- Пользователи видят "Connection error"
- Partial availability

🎯 Что тестируем:
1. Service mesh resilience
2. Circuit breaker patterns
3. Fallback mechanisms
4. DNS resolution
```

**Демонстрация:**
1. **Разорвать сеть:** `tc qdisc add dev eth0 root netem delay 5000ms`
2. **Показать:** как frontend переключается на fallback
3. **Восстановить** и посмотреть recovery

#### Слайд 14: Тушение пожара (Network Partition)
```
🔥 Что сломалось:
- 50% пользователей видят ошибки
- API calls timeout
- Business impact высокий

🚒 Как тушим:
1. **Immediate:** Circuit breakers срабатывают
2. **Short-term:** Traffic rerouting
3. **Long-term:** Multi-region deployment

📝 Lessons learned:
- Circuit breakers спасают
- Нужен traffic shadowing
- DNS failover критически важен
```

### **Часть 4: Итоги и Q&A (5-10 минут)**

#### Слайд 15: Итоги
```
🎯 Что мы узнали:

1. **Database failure:** Read-only mode + failover = 2 минуты recovery
2. **API overload:** Rate limiting спасает backend, но UX страдает
3. **Analytics failure:** Caching критически важен для business users
4. **Network partition:** Circuit breakers - must have для микросервисов

📊 Результаты:
- Recovery Time Objective: 2-5 минут
- Impact on users: Минимизирован через degradation
- Confidence level: Высокий для Black Friday
```
